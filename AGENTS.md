# AGENTS.md

## Required Reading

Before doing any work in this repository, always read:

1. `PROJECT_CONTEXT.md`
2. `docs/context_full.md`

Do not start coding, refactoring, or reviewing until both files have been read.

---

## Project Rules

This is an HRC 2026 robotics project with a fixed two-machine workflow:

- Mac = development machine
- Linux = runtime machine

Mac is used for:
- coding
- reading the repo
- using Codex for analysis and review
- lightweight tests
- commit and push

Linux is used for:
- simulation
- evaluation
- batch tests
- benchmarks
- logs
- outputs
- dataset
- checkpoints

Linux is the runtime source of truth.

---

## Core Workflow

The workflow must remain:

1. Edit on Mac
2. Push via Git
3. Pull on Linux
4. Run on Linux
5. Inspect logs and metrics
6. Iterate again on Mac

Do not change this workflow unless there is a strong technical reason.

---

## Scope of Changes

When making changes:

- keep changes minimal
- keep changes local to the task
- do not refactor unrelated files
- do not redesign project structure without explicit justification
- do not introduce broad architecture changes early

Prefer small, testable patches over large rewrites.

---

## Engineering Priorities

Always prioritize in this order:

1. correctness with respect to project rules
2. runtime stability
3. simplicity and debuggability
4. reproducibility
5. performance optimization

Rule of thumb:

> baseline stable first, optimize later

---

## Path and Environment Rules

Never hardcode local paths such as:

- `/Users/...`
- `/home/...`

Always use environment variables or config mapping.

Expected Linux environment variables include:

- `HRC_ROOT`
- `HRC_REPO`
- `DATA_ROOT`
- `CKPT_ROOT`
- `OUTPUT_ROOT`
- `LOG_ROOT`

Code should remain path-agnostic and portable between Mac and Linux.

---

## Git and Repository Hygiene

Never commit:

- dataset
- checkpoints
- outputs
- logs
- videos
- replay files
- images generated by runs
- caches
- build artifacts
- virtual environments

Respect `.gitignore`.

If a directory must exist in Git while empty, use `.gitkeep`.

---

## Testing Rules

On Mac:
- only run lightweight tests
- smoke tests are allowed
- do not treat Mac results as final runtime truth

On Linux:
- run actual tests
- run simulation
- run evaluation
- run benchmark and robustness checks

Heavy testing belongs on Linux only.

---

## Simulator Rules

If the project uses Isaac Sim or other heavy simulator dependencies:

- keep simulator installation outside the repo
- do not silently change simulator version
- document simulator version changes in `PROJECT_CONTEXT.md` or `TASK_LOG.md`

---

## Codex Behavior Rules

When acting as Codex in this repository:

- read `PROJECT_CONTEXT.md` first
- read `docs/context_full.md` for full system understanding
- check existing files before creating new ones
- prefer editing existing files over creating duplicates
- do not invent missing infrastructure unless necessary
- do not add heavy dependencies unless explicitly required
- do not assume the Mac environment is equivalent to Linux runtime
- do not assume dataset is available inside the repo

If something is uncertain, make the smallest safe assumption and state it clearly.

---

## Logging and Documentation

After meaningful changes:

- update `TASK_LOG.md`
- record what changed
- record why it changed
- record test result if any
- record next step if clear

Keep documentation concise but useful.

---

## Output Style for Code Changes

When proposing or applying changes:

- explain what is being changed
- explain where it is being changed
- explain why it is needed
- avoid unnecessary verbosity
- avoid unrelated cleanup

---

## Anti-Patterns

Do not:

- refactor large parts of the repo early
- hardcode machine-specific paths
- move runtime assets into Git
- treat Mac-only tests as final validation
- optimize before a stable baseline exists
- create nested repos
- introduce broad abstractions without immediate need

---

## Current Development Philosophy

This project is still in the baseline-building stage unless explicitly updated otherwise.

Default approach:

- deterministic baseline first
- minimal moving parts
- easy rollback
- easy debugging
- clear logs
